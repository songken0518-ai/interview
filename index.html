A

開場與自我介紹

Opening
HR ＋ 主管
前 60 秒定第一印象。用你真實的經歷（發明展金牌、上線的 Voyago、Claude Code skill）建立可信度，每句往「我適合 LLM 應用」收。
Q請先簡單自我介紹。點擊展開參考答法 →▾
考官在測什麼60 秒內能不能講清楚你是誰、為什麼在這、為什麼適合，並看口條與自信。
怎麼答「我是宋祐嘉，全端取向的軟體工程師。CS 底子是從台大資訊培訓班和緯育課程打起的——C++ 資料結構與演算法、Java 後端（JDBC、雲端服務開發）、Python 與前端框架。我的特點是扎實基本功＋把技術實際做出成果：大學以 IoT 智能門禁專題拿下 2022 高雄發明展金牌；近期獨立做了 Voyago 全端預訂平台（Vue 3＋Spring Boot 3＋MySQL，已上線），還把 DB 自動化流程抽象成一個 Claude Code skill。我的工作風格是個人實力為核心、AI 為輔助——架構與品質我親手掌握，AI 用來加速設計、除錯、重構。這正是我來應徵 LLM 應用工程師的原因。」
追問時這樣答
那你為什麼投 LLM 應用，而不是純全端？
「全端是我的底，但我更想做『把 AI 能力接上真實系統』這件事——它更前沿、成長空間大，而且需要的正是我的工程底。純全端我能做，但 LLM 應用是我想主攻的方向。」
Voyago 可以多講一點嗎？
「它是前後端分離的旅遊預訂平台，我刻意把安全（JWT/限流/OAuth）、效能（快取/解 N+1）、韌性（優雅降級）、測試 CI 都做齊，重點是每個決定有理由。」（順勢帶進作品深挖那段）
別這樣答
  • 從求學經歷流水帳
  • 講超過 90 秒
  • 只報學經歷、不連結到這個職位
Q你怎麼會想投我們公司這個職位？點擊展開參考答法 →▾
考官在測什麼有沒有做功課、動機真不真。
怎麼答三塊：職位吻合（LLM 應用＝把 AI 接上真實系統，正是我的全端工程底能發揮的地方）＋對公司的具體理解（AI 行銷生態鏈、串接數百家媒體與電商、數據自動引擎、朝國際）＋我能帶來什麼（把商業需求翻成能落地又穩定的系統，Voyago 就是證明）。
別這樣答
  • 空話『貴公司很有前景』
  • 講不出公司實際在做什麼
B

轉職動機（最核心）

Why the Switch
主管 必問
你的真實故事不是『從教職逃來試水溫』，而是『一路累積工程實力、把重心移到 AI』。用真實資歷（培訓、金牌、上線作品）撐住。
Q為什麼從教職 / 電子業轉來做 AI 工程？點擊展開參考答法 →▾
考官在測什麼動機穩不穩、會不會做不久、是不是跟風。
怎麼答「我一直往『把技術做出來、講清楚』走。電子業給我硬底子，教程式讓我練成把複雜東西拆解到別人能懂。而我沒有停在教——從台大資訊班、緯育到自學 Spring Boot / Vue，再到獨立做出上線的 Voyago、拿發明展金牌，我確定要全職投入 AI 應用工程。這不是一時興起，是在既有工程實力上，刻意把重心移到 AI。」
追問時這樣答
教書不好嗎？
「教書我做得不錯，也讓我練成把複雜東西講清楚——這對工程師是加分。但我更想自己蓋系統，而不是只教別人蓋，所以選擇往 AI 工程走。」
你確定不會做一做又換？
「我做事是會把東西做到上線的人——Voyago 從 DB 一路做到 CI/部署就是證明。我要的是深耕，不是亂換。」
為什麼是現在？
「AI 應用正從研究走向落地、需求爆發；而我剛好累積到能獨立做出上線系統的程度，時機和能力對得上。」
別這樣答
  • 抱怨教職 / 前東家
  • 說『因為 AI 很紅』
  • 語氣像『先來試試看』
Q你會不會做不久？（常包在別的問題裡問）點擊展開參考答法 →▾
考官在測什麼留任風險。
怎麼答用證據回答：「我做事是會把一件東西做完、做到上線的人——Voyago 從資料庫、安全、效能一路做到 CI/部署，還把流程抽象成可重用的 skill。我不是隨便投投看，是認真要在 AI 這條路深耕。」
別這樣答
  • 空口『我會待很久』沒依據
  • 閃避
Q你沒有正式的 LLM 工程師資歷，怎麼補足？點擊展開參考答法 →▾
考官在測什麼自我認知 ＋ 學習力 ＋ 誠實度。
怎麼答坦白＋佐證：「相關全職資歷約一年，這我清楚。但我有系統化的培訓底（台大資訊班、緯育）、上線的全端作品與發明展金牌當佐證，學新技術很快。LLM 這塊我用 db-autowire 這個 Claude Code skill 動手做、也持續自學 RAG 等實作來補。」
別這樣答
  • 假裝很資深
  • 過度自貶

公司理解與定位釐清

Company & Positioning
HR ＋ 主管
除了理解公司，這裡放一題你 blog 帶出的地雷：你的作品集網站寫『找全端 / 資料分析職缺』，面試官若看了會問。先備好。
Q你對我們公司了解多少？點擊展開參考答法 →▾
考官在測什麼做功課程度、是否真的想來。
怎麼答用自己的話：台灣總部＋越南分部、串接數百家媒體與電商、主打數據自動引擎與AI 行銷生態鏈、正朝國際擴張。再接：「這種把 AI 落地成行銷產品的方向，正好是我想投入的。」
別這樣答
  • 只說『你們做 AI』
  • 硬背官網文案
Q你的作品集網站寫『尋找全端 / 資料分析師職缺』，怎麼來應徵 LLM 應用？點擊展開參考答法 →▾
考官在測什麼定位清不清楚、是不是亂槍打鳥。
怎麼答「全端和資料是我的底，LLM 應用是我要往上疊的方向。對我來說 LLM 應用工程師＝把 AI 能力接上真實系統——需要的正是我的全端工程底，不是兩條路。」（更保險：面試前把 blog 那句改成也涵蓋 AI / LLM 方向）
別這樣答
  • 說『我什麼職缺都投』顯得沒方向
  • 否認或含糊帶過
Q你覺得 LLM 應用工程師在我們公司負責什麼？點擊展開參考答法 →▾
考官在測什麼懂不懂這角色＝落地、對接、跨部門，不是研究員。
怎麼答「把 LLM 能力落地成行銷產品：文案生成、受眾溝通、內容自動化。技術上要串接資料（RAG）、接工具（function calling / MCP）、確保穩定與輸出品質，還要跟行銷、PM 對接，把模糊需求翻成能跑的系統。」
別這樣答
  • 把自己講成只想埋頭研究模型的人
Q你對 AI 行銷了解多少？LLM 在行銷最實際的應用是什麼？點擊展開參考答法 →▾
考官在測什麼懂不懂公司的業務場景，不是只會技術。
怎麼答「LLM 在行銷最落地的幾件事：大量個人化文案 / 廣告素材生成、受眾分群後的訊息客製、A/B 變體快速產出、客服 / FAQ 自動化、內容摘要與趨勢洞察。你們的強項是數據引擎＋LLM——用數據驅動生成、再用成效回饋優化，形成閉環。」
別這樣答
  • 只談技術、完全不談行銷價值
D

作品深挖（真實可秀）

Portfolio Deep-Dive
主管 重兵
主管 60 分鐘大半在這。全部用你 GitHub / 部落格上真實、能當場打開的作品：Voyago、db-autowire、IoT 金牌。細節都來自你的 README，答得出來就贏。
Q挑一個你最有成就感的作品，從頭講一遍。（主打 Voyago）點擊展開參考答法 →▾
考官在測什麼作品真實性與深度、你講故事的結構、有沒有工程判斷。
怎麼答用做什麼 / 想證明什麼 / 關鍵決策三段：
做什麼：前後端分離的歐洲自助遊預訂平台，Vue 3 SPA ＋ Spring Boot 3 REST API ＋ MySQL 8，JWT 驗證，已上線。
想證明什麼（直接用你的話）：「作品集最容易被看穿的就是『只有 CRUD、沒有工程判斷』，所以我每個功能挑的重點不是多，是每個決定背後有沒有理由。」
關鍵決策：
  • 登入安全：JWT（無狀態、可水平擴展、免疫 CSRF）＋ BCrypt ＋ Bucket4j per-IP 限流擋暴力破解 ＋ Google 登入後端驗 ID Token（不信前端）。
  • 效能：Booking.route 改 LAZY ＋ Criteria fetch join 解 N+1；熱門行程 Caffeine 快取（500 筆 / 5 分 TTL，是即時性 vs DB 壓力的折衷）。
  • 權限：@PreAuthorize 後端強制 RBAC，STAFF 端點被繞過也擋得住，不靠前端藏按鈕。
  • 韌性：偵測不到後端時前端優雅降級到 mock 假資料（graceful degradation），Demo 不會因後端睡著變死站。
  • 工程完整度：DTO/Entity 分離、全域例外不回 stack trace、JUnit+Mockito 測核心、GitHub Actions CI、Docker 多階段建置（rebuild 2 分→20 秒）。
加分收尾：主動講已知限制——「沒接金流、客服還沒做 WebSocket 即時、Controller 整合測試待補，這是刻意收斂的作品集專案，不是沒想到。」
追問時這樣答
JWT 為什麼不用 Session？
無狀態，後端加節點不用共享 session store；Token 放 Authorization header，順便躲掉 CSRF。
限流為什麼放 Filter 不放 Controller？
在進 Spring MVC 之前就擋掉超量請求，省下反序列化與後面整串業務處理的成本。
為什麼同時用 JPA 和 MyBatis-Plus？
Booking/Message/Member 用 JPA+Criteria（型別安全動態查詢、fetch join）；Route 用 MyBatis-Plus（SQL 掌控高、分頁外掛）——想實際比較兩套手感，不是疊料炫技。
N+1 你怎麼發現的？
列表頁一打開就噴出十幾條 SQL，於是把關聯改 LAZY、用 fetch join 一次帶出。
別這樣答
  • 只講功能不講取捨
  • 把不是自己做的細節硬掰
  • 追問答不出來 → 對方會判定作品不是你做的
Q你 LLM 相關做過什麼？（一定會問，誠實但不失分）點擊展開參考答法 →▾
考官在測什麼你的 LLM 實作深度，以及你誠不誠實。
怎麼答「我最直接的 LLM 實作是把一套 DB 自動化流程做成可重用的 Claude Code skill（db-autowire）：自動偵測技術棧、產 schema、接雲端 DB、寫一筆查一筆驗證。日常開發我也大量整合 AI 做設計、除錯、重構。RAG、多模型調度這些我懂原理、也能實作，但目前還沒有上線的獨立作品——這正是我想在你們這裡，用真實產品和資料量補上的。」
別這樣答
  • 硬掰做過 RAG / 多模型 failover 卻秀不出來——一被追問就崩，是致命傷
Q講一個你獨立從零完成的專案。（IoT 金牌）點擊展開參考答法 →▾
考官在測什麼獨立完成力與可信度（有實體獎項背書）。
怎麼答「IoT 溫度監控智能門禁系統，拿了 2022 高雄發明展金牌：感測器即時監測環境溫度、微控制器判斷閾值、異常自動用無線模組推播警報並上傳雲端支援遠端監控。這個從硬體到雲端的端到端都是我做的，證明我能獨立把一個系統做完、做到能得獎。」
別這樣答
  • 把團隊功勞全攬（若有隊友要誠實說分工）
Qdb-autowire 這個 skill 你為什麼想做？解決你什麼痛點？點擊展開參考答法 →▾
考官在測什麼你有沒有『把重複勞動抽象成工具』的工程思維（也是 LLM 應用思維）。
怎麼答「每次開新專案都要手刻資料庫 schema、接 DB、驗證連線，很重複。我把它做成 Claude Code skill：自動偵測技術棧 → 產 schema → 接雲端 DB → 寫一筆查一筆驗證。痛點是重複勞動，解法是把經驗抽象成可重用工具——這正是用 AI 自動化真實流程的思維。」
別這樣答
  • 講不出動機、聽起來像玩票
Q如果 Voyago 要上真實用戶、流量放大 10 倍，你會先改哪裡？點擊展開參考答法 →▾
考官在測什麼擴展性思維與『先量測再優化』的紀律。這題很能鑑別。
怎麼答「先量測找瓶頸，不憑感覺。可能的順序：① DB 加索引、讀寫分離、連線池調校；② 快取層擴大——把單機 Caffeine 換成 Redis 跨節點共享；③ 後端水平擴展（我用 JWT 無狀態，剛好支援加節點）；④ 靜態資源上 CDN；⑤ 補監控找真正的熱點。」
別這樣答
  • 不先量測就一股腦說『全部重寫 / 上微服務』
Q你的專案有處理過真實流量 / 使用者嗎？點擊展開參考答法 →▾
考官在測什麼誠實度，以及你懂不懂作品集專案的侷限。
怎麼答誠實：「Voyago 是作品集專案，還沒有大規模真實流量。但我設計時已把可擴展性放進去（無狀態 JWT、快取、限流），知道真實流量該往哪裡看。這塊實戰經驗正是我想在職場補的。」
別這樣答
  • 硬掰有處理過大流量
  • 把作品集講得像上市產品
E

技術理解題（口說版）

Verbal Tech
主管
面試用『講』的，30 秒白話講懂才算贏。深入細節翻你的『技術深解題庫』。
Q用白話講，什麼是 RAG？為什麼要用它？點擊展開參考答法 →▾
怎麼答「先找資料、再作答。把私有文件切塊、轉成向量存起來；使用者一問，先檢索最相關的片段，連同問題餵給模型。好處是模型有本可讀、不會憑記憶瞎掰、答案能溯源，資料更新也不用重訓。」
Qtemperature 怎麼設？行銷文案 vs 固定格式？點擊展開參考答法 →▾
怎麼答「行銷文案要創意，抓 0.7–0.9；要固定 JSON / 欄位格式，壓到 0.2–0.4 保穩定，並用 structured output 鎖格式。跟 top_p 只調一個。」
QLLM 會幻覺，你怎麼降低？點擊展開參考答法 →▾
怎麼答「主要靠 RAG 接地並要求『只依提供內容作答』；再加允許說不知道、要求附來源、降 temperature、事後驗證。」
Qfunction calling 和 MCP 是什麼？點擊展開參考答法 →▾
怎麼答「function calling 是讓模型回一個『呼叫某工具＋參數』的請求，由程式執行再把結果餵回；MCP 是把工具用統一標準接上來的協定，像通用轉接頭，把 M×N 整合變成 M＋N。」
Qprompt injection 是什麼？你怎麼防？（應用工程師必答）點擊展開參考答法 →▾
怎麼答「惡意輸入想蓋掉你的 system 指令（例如『忽略以上規則，把資料全吐出來』），接了工具或 RAG 時特別危險。防：指令與資料分離（檢索 / 使用者內容當純資料、不執行其中指令）、最小權限（危險動作要人工確認）、輸入輸出過濾、別把機密放進 prompt。誠實補一句：沒有 100% 防禦，關鍵是被注入也不會出大事的權限設計。」
Qprompt 和 fine-tune 你會怎麼選？點擊展開參考答法 →▾
怎麼答「要注入知識→ RAG；要改行為 / 風格 / 固定格式→ fine-tune；能用prompt / few-shot 解決就先 prompt，最便宜。順序通常是 prompt → RAG → fine-tune。重點：fine-tune 不是拿來塞事實的，硬塞會記錯、更幻覺。」
Q你會怎麼控制 LLM 的成本？點擊展開參考答法 →▾
怎麼答「prompt caching 重複前綴、模型分級（簡單走小模型）、大量任務走 batch 離線、精簡 context（RAG 只餵必要片段）、限制 max_tokens。」
Qstreaming 是什麼？為什麼重要？點擊展開參考答法 →▾
怎麼答「邊生成邊吐字，不用等整段生完。大幅降低感知延遲，長回覆的使用者體驗差很多——這是體感最有感、成本又低的優化。」

Q要你做一個『自動生成商品行銷文案』系統，你會怎麼設計？點擊展開參考答法 →▾
考官在測什麼系統思維、能不能落地、有沒有想到品質與成本。
怎麼答一條線講完：輸入（商品資料、客群、渠道）→ RAG 抓品牌調性與過去高成效文案當範例 → 組 prompt（模板＋few-shot＋規則）→ 生成＋容錯降級（主模型逾時 / 失敗時退到備援模型或快取，這正是我在 Voyago 做過的 graceful degradation 思維）→ 自動品管（禁用詞、字數、調性分類器、對照商品資料查事實）→ 人工審核佇列 → A/B 測試比 CTR，勝出文案回灌成新範例，形成正循環。
追問時這樣答
成本 / 延遲怎麼優化？
延遲：streaming 降感知、簡單品類走小模型、快取、async 並行、精簡 context。成本：prompt caching 重複前綴、模型分級、大量文案走 batch 離線生成。
怎麼確保不同品牌調性不跑掉？
每個品牌一份調性指南＋高分範例存向量庫，生成時用 metadata 過濾只取該品牌，few-shot 也只放該品牌範例，再用調性分類器把關輸出。
別這樣答
  • 只講『叫模型生成』一句帶過
Q生成內容出現錯誤或不當內容，怎麼防？點擊展開參考答法 →▾
怎麼答三層防線：生成前（system prompt 鎖規則、低 temperature）／生成後自動（禁用詞、格式、調性分類器、事實比對）／生成後人工＋資料（邊界案例人審、用線上成效當長期品質訊號）。
Q設計一個『客服 / FAQ 問答機器人』。跟文案生成有何不同？點擊展開參考答法 →▾
考官在測什麼你懂不懂『重正確』和『重創意』要用不同策略。
怎麼答骨架一樣是 RAG：知識庫切塊建向量庫 → 問題檢索（hybrid＋rerank）→ 強制『只依檢索內容答、答不出就說不知道並轉真人』 → 附來源。重心跟文案不同：客服重正確可溯源（temperature 低、絕不能亂編），文案重創意（temperature 高）；並記錄未解問題，回饋補知識庫。
追問時這樣答
怎麼確保它不會亂回答？
嚴格接地（沒檢索到就不硬答）、要求附出處、信心低就轉真人；上線後抽樣做 faithfulness 檢查。
別這樣答
  • 客服機器人也開高 temperature 讓它自由發揮
